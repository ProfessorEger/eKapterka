# Data Model

This document describes the Firestore and storage data model implemented in the current codebase.

## 1. Persistence Overview

Primary stores:

- Firestore: structured domain records (`categories`, `items`, `rentals`, `users`).
- Google Cloud Storage: binary item photos.

Code sources:

- Domain structs: `internal/models/models.go`
- Firestore access: `internal/repository/*.go`
- Seed data: `internal/seed/categories.go`

## 2. Firestore Collections

## 2.1 `categories`

Document ID:

- Deterministic string category key (e.g. `equipment`, `mountain`, `carabiners`).

Fields:

- `id` (`string`): same logical ID as doc ID.
- `title` (`string`): display name.
- `parent_id` (`string | null`): parent category ID; top-level uses `root` sentinel.
- `order` (`int`): sibling sorting key.
- `is_leaf` (`bool`): true when category has no subcategories in browsing flow.

Query patterns in code:

- Get category by ID.
- Get children by `parent_id`, sorted by `order ASC`.
- Get leaf categories by `is_leaf == true`, then in-memory sort by `order`, `title`, `id`.

Behavioral notes:

- `parent_id` root convention uses constant `models.RootParentID == "root"`.
- Seed tool upserts each category with `Set`.

## 2.2 `items`

Document ID:

- Auto-generated by Firestore on create (`Collection("items").Add`).

Fields:

- `title` (`string`)
- `description` (`string`)
- `category_id` (`string`): references `categories.id`.
- `tags` (`string[]`): currently stored but not managed by commands.
- `photo_urls` (`string[]`): public GCS URLs.
- `created_at` (`timestamp`)
- `updated_at` (`timestamp`)
Read compatibility behavior:

- Legacy embedded rentals (`rentals` / `rental_periods`) are no longer read in code.

Query patterns in code:

- Get item by document ID.
- Get items by category with pagination:
  - primary query: `Where(category_id==...) + OrderBy(created_at DESC) + Offset + Limit(limit+1)`
  - fallback query: same without `OrderBy` if index/order query fails.

Mutation patterns:

- Add item: sets `created_at`, `updated_at` to `time.Now()`.
- Edit item: updates title/category/description/photo_urls and refreshes `updated_at`.
- Delete item: deletes document by ID.
- Add rental: create document in `rentals`, update `items.updated_at`.
- Remove rental: delete `rentals` document by ID, update `items.updated_at`.

## 2.3 `rentals`

Document ID:

- Auto-generated by Firestore on create.

Fields:

- `item_id` (`string`): references `items` document ID.
- `start` (`timestamp`)
- `end` (`timestamp`)
- `description` (`string`)
- `user_id` (`int64`): Telegram user ID of renter.

Query patterns in code:

- Get rentals by `item_id`.
- Get rentals by `user_id` (used in profile flow).

## 2.4 `users`

Document ID:

- Stringified Telegram user ID (`strconv.FormatInt(userID, 10)`).

Fields:

- `id` (`int64`): Telegram user ID.
- `role` (`string`): `user` or `admin`.
- `created_at` (`timestamp`)
- `message_id` (`int64`): currently initialized to 0; not actively used in flow.

Role lifecycle:

- On first `/start` or role query, missing user doc is auto-created with `role=user`.
- `/getadmin <code>` updates role to `admin` if runtime secret matches.
- `/grantadmin <user_id> <code>` updates role to `admin` for the target user (admin-only).
- `/revokeadmin <user_id> <code>` updates role to `user` for the target user (admin-only).

## 3. Cloud Storage Data Model

Bucket:

- Defined by env var `STORAGE_ID`.

Object naming:

- `items/<chat_id>/<unix_nano>_<telegram_file_unique_id>.<ext>`

Extension mapping from HTTP content type:

- `image/png` -> `.png`
- `image/webp` -> `.webp`
- `image/gif` -> `.gif`
- default -> `.jpg`

Public URL format:

- `https://storage.googleapis.com/<bucket>/<object_path>`

Deletion logic:

- On item delete, each `photo_url` is converted back to object path (supports two GCS URL host formats) and deleted.
- On edit with replaced photo, old photo objects are deleted after successful item update.

## 4. Schema Invariants and Assumptions

1. Category graph integrity is expected from seed data (no runtime validator).
2. `items.category_id` should point to a leaf category for correct UX.
3. Rental intervals are not deduplicated and may overlap (no overlap validation in code).
4. `start <= end` for rental is enforced at command parse level.
5. `rentals.item_id` should point to existing `items` documents for correct rendering.
5. User role strings are expected to match constants (`admin`, `user`).

## 5. Seeded Category Taxonomy (Current)

Root logical groups under `root`:

- `equipment`
- `inventory`

Nested examples:

- `equipment -> mountain -> hardware -> carabiners`
- `equipment -> ski -> skis`
- `inventory -> furniture`

The full authoritative list is in `internal/seed/categories.go`.

## 6. Firestore Index Considerations

Potential index requirements:

- `items` query using `category_id` filter + `created_at` order may require a composite index depending on Firestore project state.
- Code includes fallback query without order to keep functionality when ordered query fails.

## 7. Data Quality and Migration Notes

Not currently implemented:

- Explicit schema migrations.
- Referential integrity enforcement (e.g., deleting category with item references).
- Soft deletes / audit trails.

## 8. Suggested Future Hardening

1. Enforce category existence and leaf status on `/add` and `/edit`.
2. Validate rental interval overlaps depending on business rules.
3. Add optimistic concurrency controls for item updates if multi-admin edits become common.
4. Add migration scripts/versioning for schema evolution.
